§3 python-patterns

410 lines, single file · solid defaults, two real issues

medium"Anti-Patterns to Avoid" shows the corrected code, not the anti-pattern.

Every example in that section is the fixed version: the "Mutable default argument" entry shows the None-sentinel idiom, "Catch the exception you can handle" shows a correctly narrow except. A reader — or a model skimming this skill mid-task — can plausibly parse these as the patterns to avoid. Either rename the section ("Correct patterns for common pitfalls") or restructure as explicit avoid → prefer pairs, the way style-guidelines.md does it.

mediumTyping guidance stops at Python 3.10 while the sibling skill defaults to 3.12.

The skill teaches TypeVar + typing.TypeAlias, but python-tooling's template sets requires-python = ">=3.12". On 3.12+, PEP 695 syntax is the modern form (type JSON = ..., def first[T](items: Sequence[T]) -> T | None), and typing.TypeAlias is deprecated as of 3.12. Likewise datetime.UTC (3.11+) reads better than timezone.utc in the dataclass example. Add the 3.12+ forms as the default and keep the older spellings as the noted fallback for projects on earlier floors.

noteEverything else holds up.

Error-handling boundaries, logging ownership, EAFP guidance, concurrency model selection with the "unbounded concurrency is a production bug" line, and the review checklist are all accurate and well-calibrated. No other changes suggested.

§5 python-tooling

§6 Cross-cutting

mediumpython-tooling and python-testing disagree on the default pytest config.

Tooling's pyproject.toml template bakes coverage into addopts = "--cov=mypackage --cov-report=term-missing"; testing recommends addopts = ["--strict-markers", "--import-mode=importlib"] and frames coverage as deliberate policy, not a default flag. Testing has the better position — --cov in addopts runs on every invocation, interferes with debugger breakpoints, and contradicts the "coverage gates reflect risk" stance. Align tooling's template with testing's, and let one skill own pytest config canonically while the other points to it.

lowThe three Python skill descriptions don't state when to trigger.

Skill selection happens off the frontmatter description alone, and the writing-review skill shows the winning pattern ("Use for… Not for…"). The Python descriptions are noun phrases; python-patterns and python-testing keep their "use this when…" sentence in the body, where the selector never sees it. Lift the trigger conditions into the descriptions.
